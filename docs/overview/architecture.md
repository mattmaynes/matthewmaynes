# Architecture

## Stack

- **Framework:** Next.js (App Router) + TypeScript, run as a **Node server** via `output: 'standalone'`
  (chosen over static export so the contact form / route handlers / dynamic OG work without re-architecting).
- **Styling:** Tailwind CSS **v4** (CSS-first) on the **Harbor** theme - `@rogueoak/roots` tokens
  re-branded via the roots brand pipeline (DTCG sources in `brand/harbor/` compile with `npm run
  theme:build` to `src/styles/brand-harbor.generated.css`, an AA-guarded `:root` + `.dark` block).
  Harbor maps `primary` + neutrals and inherits the rest from Roots. Components read only semantic
  tokens, so light/dark re-theme with no per-component code.
- **Fonts:** self-hosted via `@fontsource-variable/{figtree,geist-mono}`.
- **Content as data:** blog posts in `content/blog/*.mdx`, projects in `content/projects/*.mdx`. No
  database, no runtime fetching.

## Blog (specs 0009/0013/0027/0034)

- A pure JS seam `src/lib/blog.js` (unit-tested, no `gray-matter`) hand-parses frontmatter for the
  cheap listing; `next-mdx-remote/rsc` `compileMDX` renders the body in a Server Component
  (`src/components/post-body.tsx`) over our own tracked files only. Inline media is resolved through
  registries that mirror the `site.ts` map: `blog-images.ts` (static-imported, blur placeholder,
  `pixelated` flag) and `blog-videos.ts` (self-hosted H.264 + poster + intrinsic dimensions). A
  `<PostImage>`/`<PostVideo>` throws on an unknown name, so a typo fails the build.
- Pure, fs-free view helpers live in `blog-view.js` (the one `slugify`, `tagSlug`/`tagFromSlug`, the
  shared `PostRowData`/`Cover` types) so BOTH the `"use client"` listing island and the Server
  Component pages import them. The listing row is a hook-free presentational `post-row.tsx` shared by
  the island and the tag archive; a server-only `post-summaries.ts` maps posts to row data (resolving
  covers + the **global** "New" badge, computed once over all posts). A `series` frontmatter field
  drives the hero **sash** (`post-article.tsx`) and the row pill - see `docs/rules/blog-series.md`.
- **Drafts:** a `draft: true` flag partitions `getAllPosts()` into `getPublishedPosts()` /
  `getDraftPosts()`. Every public enumeration (listing, home, `/subscribe`, feed, sitemap, tag pages,
  "New" badge, published nav + `generateStaticParams`) uses the published set; drafts get their own
  `noindex` routes under `/blog/drafts`. The published and draft `[slug]` shells render one shared
  `post-article.tsx` parameterised by a single `variant` discriminator, so a draft previews identically
  to how it will publish.
- **RSS:** a pure, fs-free `rss.js` builder (unit-tested) feeds a `force-static` `app/blog/feed.xml`
  route; output is deterministic (`lastBuildDate` = newest post's date, not `Date.now()`).
- Each post has a `generateStaticParams`-baked satori OG card (`app/blog/[slug]/opengraph-image.tsx`).
  Syntax highlighting (`rehype-pretty-code` + Shiki) is planned but not yet wired.

## Metadata & sharing (spec 0004)

- **Next file conventions, not hand-rolled `<head>`:** `app/{favicon.ico,icon,apple-icon}`,
  `app/opengraph-image.tsx` (+ re-exporting `twitter-image.tsx`), `app/{robots,sitemap,manifest}.ts`.
  `layout.tsx` carries the default OG/Twitter/robots metadata, `themeColor`, and a JSON-LD `Person`.
- **One source of truth:** identity/social/`nav` come from `src/lib/site.ts`; the sitemap enumerates
  nav routes + every post + every tag archive. Icons are generated by `scripts/build-icons.ts` (`sips`
  + a stdlib ICO writer, no ImageMagick). The OG card loads woff fonts (satori can't read woff2) via
  `new URL(.., import.meta.url)` so they trace into the standalone build.

## Endpoints (specs 0008/0018/0032)

- **`POST /v1/contact`** and **`POST /v1/subscribe`** are thin HTTP shells over pure, fs-free cores
  (`contact.js`, `subscribe.js`) unit-tested without a server; the route only maps request/env/outcomes
  to status codes (400/403/413/429/500, honeypot → silent 200). Shared spam guards (honeypot, validation,
  per-IP rate limit, same-origin) live in `http-guards.js`, imported by both. Secrets are env-only, so
  they can't reach the bundle or repo (guarded structurally).
- **Contact** sends an on-brand HTML notification via Resend (`fetch`, no SDK) rendered from
  `emails/templates/contact-notification.html` (bundled via `outputFileTracingIncludes`, fields HTML-
  escaped), and records the sender in CTCT. **Subscribe** mints a 24h token from the non-rotating CTCT
  refresh token, cached in module scope with **in-flight-mint dedup** and a **401 self-heal** (clear +
  re-mint + retry once); the add hits the create-or-update `sign_up_form` endpoint (idempotent) and
  throws status-only (never the body, which can echo the email).

## Analytics & observability (spec 0014/0016)

- **PostHog (US Cloud):** `posthog-js` (client) + `posthog-node` (server). One config seam
  `src/lib/analytics.ts` (also imported by `next.config.ts` for the `/ingest` rewrites). Client init is
  module-scope + idempotent (not in an effect, so the first `$pageview` isn't dropped); replay masks
  all inputs and the contact form is additionally `ph-no-capture`. Server errors flow through
  `instrumentation.ts` `onRequestError` → a `flushAt:1` singleton; `error.tsx`/`global-error.tsx`
  re-apply the theme and route a `ChunkLoadError` through `chunk-recovery.ts`.
- **Same-origin proxy:** `/ingest/*` rewrites to `*.i.posthog.com` (with `skipTrailingSlashRedirect`),
  so ad-blockers/CSP need no third-party exception. **Local never captures** - a pure `analytics-env.js`
  seam gates the client on production + non-local hostname and the server on an explicit deploy-only
  `POSTHOG_SERVER_CAPTURE=1`, so a local production build never reaches the live project.

## Deployment (specs 0006/0010/0015/0019)

- **Container:** multi-stage Dockerfile (deps → build → runtime) on `node:24-alpine` serving the
  standalone output, built off-host and pulled from GHCR (`ghcr.io/mattmaynes/matthewmaynes`), tagged
  `latest` + `sha-<commit>`. **Host:** a small Linux VM running Compose stacks on a shared `edge`
  network; **Caddy** (owned by this repo) terminates 80/443 with Let's Encrypt and reverse-proxies by
  hostname to the site (and the cohosted rogueoak.com). Both deploy pipelines are kept symmetric.
- **CI/CD** (`deploy.yml`): push to `main` → verify (lint/build/test) → build+push to GHCR → SSH deploy
  (ensure `edge` + Caddy, `compose pull`, **zero-downtime blue/green rollout** via pinned `docker-rollout`)
  → **prewarm**. The rollout scales the site to two instances, waits for the new HEALTHCHECK, then removes
  the old; Caddy follows via **dynamic A-record upstreams** re-resolving Docker DNS. A broken image never
  goes healthy, so a bad build can't take the site down.
- **Capacity:** the blue/green overlap runs two site servers **plus** Caddy and rogueoak, so the VM is
  sized for the peak (~1GB + swap) with `mem_limit` per stack and a `timeout-minutes` on the deploy job -
  a topology change (instance count/memory) is a capacity change. A daily `cleanup-images.yml` keeps the
  10 most recent tagged GHCR images (referrer-aware, so attestation children aren't orphaned).

## Configuration & secrets

- The repo is **public**: no secrets, PII, or real contact details in tracked files or history.
- Env (all server-only, provided at runtime, `.env.local` locally): `SITE_URL`; contact
  (`RESEND_API_KEY`, `CONTACT_TO_EMAIL`, `CONTACT_FROM_EMAIL`); CTCT (`CTCT_CLIENT_ID`,
  `CTCT_REFRESH_TOKEN`, `CTCT_LIST_ID`, `CTCT_WEBSITE_LIST_ID`) powering both subscribe and the contact
  record/opt-in. On the host they live in a git-ignored `deploy/docker/.env.site` (survives the deploy's
  `git reset --hard`).
- **Exception:** `NEXT_PUBLIC_POSTHOG_*` is **inlined by `next build`**, not read at runtime, so the
  "config-free image" rule doesn't apply - the key is PostHog's publishable client token, carried as a
  committed default so CI/Docker need no new secret.

## Repo layout

- `src/app/` routes/layouts · `src/components/` UI · `src/lib/` content loaders + scrubbed `resume.ts`
  · `src/styles/` `globals.css` + generated Harbor palette + print-only `theme-harbor.css`.
- `brand/harbor/` DTCG source (`npm run theme:build`) · `scripts/` build/authoring tools · `content/`
  authored blog/project content · `emails/` HTML emails + templates · `public/` static assets + the
  committed `resume.pdf`(+`.hash`) · `context/` **git-ignored** private notes/resume · `docs/` Spectra
  specs/plans/feedback/overview + `docs/rules/`.

## Key decisions

- **Node server over static export** - server-side contact form + future dynamic needs from day one.
- **Harbor on Roots** - re-brand purely at the token layer (AA-guarded), so the design system stays
  intact and dark mode is free.
- **Resume privacy by construction** - page + PDF built from a source that omits phone/email/exact
  address; the contact destination lives only in server env.
- **Resume PDF is a committed artifact rendered from the page** by headless system Chrome, gated on a
  source hash (CI `resume:pdf:check` is a pure hash compare) so runtime never runs a browser.
- **`outputFileTracingRoot` pinned to the project** so `output: standalone` emits `server.js` at the
  standalone root even inside a nested `.worktrees/` checkout.
